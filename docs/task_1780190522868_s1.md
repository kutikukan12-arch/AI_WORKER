## 1. YouTube Data API v3 調査結果

### 1.1 seed-collector が使用するエンドポイント一覧

| エンドポイント | 用途 | クォータコスト |
|--------------|------|------------|
| `GET /youtube/v3/search` | キーワードで動画を検索 | **100 units / call** |
| `GET /youtube/v3/videos` | 動画の統計情報取得（viewCount 等） | **1 unit / call** |
| `GET /youtube/v3/channels` | チャンネルの登録者数取得 | **1 unit / call** |

### 1.2 Search.list（検索API）

**リクエスト例:**
```
GET https://www.googleapis.com/youtube/v3/search
  ?key={YOUTUBE_API_KEY}
  &part=id,snippet
  &type=video
  &q=VTuber+コラボ
  &order=viewCount
  &publishedAfter=2026-02-28T00:00:00Z
  &maxResults=50
  &regionCode=JP
  &relevanceLanguage=ja
```

**レスポンス（各アイテム）:**
```json
{
  "kind": "youtube#searchResult",
  "id": { "kind": "youtube#video", "videoId": "abc123" },
  "snippet": {
    "publishedAt": "2026-05-01T12:00:00Z",
    "channelId": "UC_xxxxx",
    "title": "【コラボ】◯◯と一緒にゲームしてみた！",
    "description": "説明文...",
    "thumbnails": {
      "high": { "url": "https://i.ytimg.com/vi/abc123/hqdefault.jpg" }
    }
  }
}
```

**制約事項:**
- `maxResults` 上限は **50件/call**
- `order` は `viewCount` / `date` / `rating` / `relevance` から選択
- `pageToken` で次ページ取得可能（ページネーション対応必須）
- **hit動画収集に主に使用**（高再生数順で検索）
- miss動画収集には不向き（低再生数順でソートできない）

### 1.3 Videos.list（動画詳細API）

**リクエスト例:**
```
GET https://www.googleapis.com/youtube/v3/videos
  ?key={YOUTUBE_API_KEY}
  &part=statistics,contentDetails,snippet
  &id=abc123,def456,ghi789   ← カンマ区切りで最大50件
```

**レスポンス（各アイテム）:**
```json
{
  "id": "abc123",
  "snippet": {
    "publishedAt": "2026-05-01T12:00:00Z",
    "channelId": "UC_xxxxx",
    "title": "タイトル",
    "description": "説明文（フル）",
    "tags": ["VTuber", "コラボ"],
    "categoryId": "20",
    "defaultAudioLanguage": "ja"
  },
  "contentDetails": {
    "duration": "PT12M34S",
    "definition": "hd"
  },
  "statistics": {
    "viewCount": "250000",
    "likeCount": "8500",
    "commentCount": "420",
    "favoriteCount": "0"
  }
}
```

**重要ポイント:**
- `viewCount` / `likeCount` / `commentCount` の型は **文字列**（`parseInt()` 必要）
- `duration` は **ISO 8601** 形式（`PT12M34S` → 754秒 に変換が必要）
- `tags` は存在しない動画もある（`|| []` でフォールバック）
- 最大 **50件/call**（1 unit）なので非常にコスト効率が良い

### 1.4 Channels.list（チャンネルAPI）

**リクエスト例:**
```
GET https://www.googleapis.com/youtube/v3/channels
  ?key={YOUTUBE_API_KEY}
  &part=statistics,snippet
  &id=UC_xxxxx,UC_yyyyy   ← カンマ区切りで最大50件
```

**レスポンス（各アイテム）:**
```json
{
  "id": "UC_xxxxx",
  "snippet": {
    "title": "チャンネル名",
    "description": "チャンネル説明"
  },
  "statistics": {
    "subscriberCount": "85000",
    "videoCount": "342",
    "viewCount": "12500000",
    "hiddenSubscriberCount": false
  }
}
```

**重要ポイント:**
- `hiddenSubscriberCount: true` の場合は `subscriberCount` が取得不可
  → buzz_ratio を計算できないため **スキップ対象**
- 最大 **50件/call**（1 unit）

### 1.5 miss動画の収集戦略

Search API では低再生数順ソートができないため、以下の手順で代替する:

**戦略A（推奨）: チャンネルの過去動画スキャン**
1. ジャンルキーワードでヒットしたチャンネルをリストアップ（hit動画収集時に取得済み）
2. そのチャンネルの最新動画を `search?channelId=xxx&order=date` で取得
3. `videos.list` で統計を取得し、buzz_ratio < 0.5 のものを miss として収集

**戦略B（補助）: 検索上位 miss を抽出**
1. `order=relevance` で幅広く検索（hit候補と同じクエリ）
2. 取得した全動画のうち buzz_ratio < 0.5 のものを miss として分類
3. hit動画のチャンネルからの miss も合わせて収集

**採用方針:** 戦略A + Bの組み合わせ（チャンネルスキャンを優先し、不足分を戦略Bで補完）

## 2. クォータ設計

### 2.1 日次クォータと制限

| 項目 | 値 |
|------|---|
| デフォルト日次クォータ | **10,000 units** |
| Search.list | 100 units / call |
| Videos.list | 1 unit / call（50動画まで） |
| Channels.list | 1 unit / call（50チャンネルまで） |

### 2.2 1ジャンルあたりのクォータ消費見積もり

**hit動画 100件収集:**
```
Search.list: ceil(300件候補 / 50件) × 100 units = 6 calls × 100 = 600 units
Videos.list: ceil(300件 / 50) × 1 unit           = 6 calls × 1  =   6 units
Channels.list: ceil(300チャンネル / 50) × 1       = 6 calls × 1  =   6 units
小計: 612 units
```
※buzz_ratio フィルタで大量脱落するため3倍の候補が必要

**miss動画 100件収集:**
```
Search.list: ceil(200件候補 / 50件) × 100 units = 4 calls × 100 = 400 units
Videos.list: ceil(200件 / 50) × 1 unit           = 4 calls × 1  =   4 units
Channels.list: チャンネルは hit時に取得済みのため追加コストほぼ 0  =   4 units
小計: 408 units
```

**1ジャンル合計:** 約 1,020 units

**全5ジャンル合計:** 約 5,100 units（余裕込みで 7,000 units 想定）

### 2.3 収集スケジュール（クォータ内に収める設計）

```
1日目: VTuber ジャンル収集   (~1,020 units)
2日目: clip ジャンル収集     (~1,020 units)
3日目: music ジャンル収集    (~1,020 units)
4日目: realshot ジャンル収集 (~1,020 units)
5日目: gaming ジャンル収集   (~1,020 units)
```

→ 各日の消費が 10,000 units の約10%なので安全マージンを確保

### 2.4 クォータ超過時の安全装置

- 収集開始前に **残クォータ推定チェック**（簡易カウンタで管理）
- 推定クォータが不足する場合は **ジャンル単位で処理停止**
- 途中まで収集済みのデータは保存してから停止（再開可能設計）

## 4. データフロー設計

### 4.1 hit動画収集フロー

```
collectGenreSeeds('VTuber', { hitCount: 100 })
  │
  ├─[1] GENRE_QUERIES['VTuber'].hitKeywords をループ
  │      ↓
  │      ytSearch({ q: keyword, order: 'viewCount', maxResults: 50 })
  │      → [{ videoId, channelId, title, thumbnailUrl, publishedAt }] × 50件
  │      → nextPageToken があれば繰り返し（候補合計 ~300件 収集）
  │
  ├─[2] 重複 videoId を除去
  │
  ├─[3] ytVideosList(videoIds) — 50件ずつバッチ処理
  │      → statistics: { viewCount, likeCount, commentCount }
  │      → contentDetails: { duration }
  │      → snippet: { tags, description }
  │
  ├─[4] ytChannelsList(channelIds) — 50件ずつバッチ処理
  │      → statistics: { subscriberCount, hiddenSubscriberCount }
  │
  ├─[5] buzzRatio = viewCount / subscriberCount を計算
  │      → hiddenSubscriberCount = true はスキップ
  │
  ├─[6] filterHit(): buzzRatio >= 3.0 のみ残す
  │      → 上位 hitCount 件を選択
  │
  ├─[7] toSeedVideo() で SeedVideo 型に変換
  │
  └─[8] saveSeeds('VTuber', 'hit', videos)
         → data/diagnostic/seeds/VTuber/hit/{videoId}.json に保存
```

### 4.2 miss動画収集フロー

```
collectGenreSeeds('VTuber', { missCount: 100 })
  │
  ├─[1] hit動画収集で取得済みチャンネルID を利用（追加Search不要）
  │      ↓
  │      各チャンネルについて:
  │      ytSearch({ channelId: xxx, order: 'date', maxResults: 50 })
  │      → チャンネルの最近の動画リスト
  │
  ├─[2] ytVideosList() で統計取得
  │
  ├─[3] buzzRatio を計算
  │
  ├─[4] filterMiss(): buzzRatio <= 0.5 のみ残す
  │      → 不足分は GENRE_QUERIES[genre].missKeywords で補完検索
  │
  ├─[5] missCount 件になるまで収集
  │
  └─[6] saveSeeds('VTuber', 'miss', videos)
```

## 5. エラーハンドリング設計

### 5.1 エラー種別と対応方針

| エラー種別 | 対応 |
|-----------|------|
| `quotaExceeded` (403) | 即座に停止・保存済みデータを維持・次回再開可能 |
| `keyInvalid` (400) | 起動時チェックで事前検出・Discord に通知 |
| `forbidden` (403) | 動画単位でスキップ・ログに記録 |
| ネットワークタイムアウト | 3回リトライ（指数バックオフ: 1s, 2s, 4s） |
| `hiddenSubscriberCount` | その動画をスキップ（buzz_ratio 計算不可） |
| `statistics` 未取得 | その動画をスキップ |
| `pageToken` 取得失敗 | ページネーション停止・収集済みで処理継続 |

### 5.2 リトライ実装

```javascript
async function withRetry(fn, maxRetries = 3) {
  for (let i = 0; i < maxRetries; i++) {
    try {
      return await fn();
    } catch (err) {
      if (err.code === 403 && err.errors?.[0]?.reason === 'quotaExceeded') {
        throw err;  // クォータ超過はリトライしない
      }
      if (i === maxRetries - 1) throw err;
      await sleep(Math.pow(2, i) * 1000);
    }
  }
}
```

## 6. 実装時の注意点

### 6.1 環境変数

```env
YOUTUBE_API_KEY=    # Google Cloud Console で発行した YouTube Data API v3 キー
```

`bot/utils/seed-collector.js` の起動時に `YOUTUBE_API_KEY` 未設定の場合は
エラーをスローし、収集を実行しない（API キーなしで `require()` 可能にする）。

### 6.2 依存パッケージ

**追加パッケージ不要。** `fetch()` (Node.js 18+ 標準) のみ使用。
`node-fetch` や `googleapis` SDK は導入しない（最小依存）。

### 6.3 クォータカウンタの管理

収集セッション中に `data/diagnostic/seeds/_quota_log.json` を更新し、
当日の消費 units を記録する。日付が変わった場合はリセット。

```json
{
  "date": "2026-05-31",
  "totalUnitsUsed": 1020,
  "lastUpdated": "2026-05-31T11:30:00+09:00"
}
```

### 6.4 既存データの扱い

- `forceRefresh: false`（デフォルト）の場合: 既にファイルが存在する videoId はスキップ
- `forceRefresh: true` の場合: 既存ファイルを上書き

### 6.5 buzz_ratio の計算注意

- `subscriberCount` は文字列型 → `parseInt()` 必要
- 登録者数 0 は `Infinity` になるため除外（`subscriberCount < 100` を下限に）
- 非常に小さいチャンネル（1000人未満）は外れ値になりやすいため **5,000人以上** を推奨下限とする

```javascript
const MIN_CHANNEL_SUBSCRIBERS = 5000;

function calcBuzzRatio(viewCount, subscriberCount) {
  if (subscriberCount < MIN_CHANNEL_SUBSCRIBERS) return null;  // 除外
  return viewCount / subscriberCount;
}
```
