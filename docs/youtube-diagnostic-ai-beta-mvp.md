## 0. このドキュメントの目的

「作れる仕様」ではなく **「売れる可能性を検証できる最小商品仕様」** を確定する。
βのゴールは完成度ではなく **需要の検証**：「他人に渡して使われ、もう一度使いたくなるか」を最小コストで測る。

### 商品の一言定義
> **「再生数を当てるAI」ではなく「投稿前の成功確率を上げる診断AI」**

---

## 1. 商品コンセプト（確定）

- 投稿予定の動画メタデータ（タイトル・ジャンル等）を入力すると、**投稿前の最適化度を6軸＋総合スコアで診断**し、**改善提案TOP2と根拠**を返す。
- 診断は **YouTube無料APIで収集した実データで学習した軽量モデルをローカル推論**して行う。**診断時のAPIコストは¥0**。
- 再生数の予測・保証・レンジは **行わない**（誇大広告リスク回避・コスト方針の両面）。
- **提供形態：外部公開の Webアプリ（PWA）。** URLを開くだけで使え、インストール不要。**社内 AI_WORKER（Discord）とは完全分離**し、外部ユーザーにDiscordは公開しない（CEO/副社長 2026-06-04 決定）。
- **β目的は「完成品販売」ではなく「需要確認」。MVP肥大化を禁止する**（§3の除外リストを厳守）。

---

## 2. β版に入れるもの（MUST／CEO確定リスト）

βは **WebアプリのUI層** に下記だけを実装する。**診断エンジン（`youtube-predictor.js` 等）の仕様には触れない**——UIはエンジンの出力を表示するだけ。

### 2-1. βに含めるUI（CEO確定・これ以上増やさない）
| ✅ 含める | 内容（UI層の責務） |
|---|---|
| **タイトル入力** | 1行テキスト入力（必須） |
| **ジャンル選択** | セレクト（VTuber/切り抜き/音楽/実写/ゲーム実況/その他）（必須） |
| **診断実行** | ボタン1つ → エンジン呼び出し → 結果表示 |
| **6軸スコア表示** | CTR/視聴維持/SEO/感情フック/投稿タイミング/競合差別化（★＋数値） |
| **改善提案表示** | エンジンが返す改善提案TOP2（理由＋対策つき）をそのまま表示 |
| **フィードバック取得** | 結果画面に簡易フィードバック（例：役に立った？ / 自由記述1欄）＝需要確認の計測 |

### 2-2. 結果表示に必ず含める付帯要素（新機能ではなく表示義務）
- **総合スコア＋ランク**（エンジン出力）
- **改善提案の根拠**（理由＋対策。エンジン出力をそのまま）
- **免責表示**：「本診断は投稿前の最適化度を示すもので、再生回数を予測・保証するものではありません」を常時表示

> 補足：タグ・尺・投稿予定時間などの**追加入力はβ必須にしない**（肥大化防止）。エンジンは受け付けられるが、βのUIは「タイトル＋ジャンル」を必須とし、それ以外は任意・最小に留める。
> 既存 `diagnosePrePub()` / `buildDiagnosisSummary()` の出力をWeb UIで表示するだけ。**UI層のみの作業。**

## 3. β版で禁止するもの（MUST NOT）

### 3-1. UI／機能の除外（CEO確定・MVP肥大化禁止）
βに**入れない**（需要確認に不要・後フェーズ送り）：

| ❌ 除外 | 理由 |
|---|---|
| **ログイン** | 需要確認に不要。匿名で触れる方が初回体験が軽い |
| **決済** | βは全機能無料。課金は需要確認後（Phase2） |
| **アカウント管理** | 同上。履歴やマイページは後フェーズ |
| **高度な分析画面**（ダッシュボード/トレンド等） | β＝最小の診断1画面に集中 |
| **SNS連携** | 不要。共有は単純なURLコピーで十分 |
| **チャンネルOAuth** | 実装重・将来の上位機能（Phase3） |

> 上記に該当する要望が出ても **βスコープに入れない**。判断に迷ったら「需要確認に必須か？」で切る。

### 3-2. 商品メッセージ上の禁止（恒久）
| 禁止 | 理由 |
|------|------|
| **再生数の保証** | 誇大広告・信頼失墜リスク |
| **再生数レンジ表示**（「○〜○万回」） | 設計思想に反する。視聴予測AIの旧出力は診断スコアへ置換 |
| **LLM(Claude/OpenAI)依存** | 診断ごとに課金が発生＝無料運営で赤字化。コトノハ方針違反 |
| **有料API必須機能**（診断時にYouTube Data API/Vision を叩く機能） | 限界コストを発生させない。サムネ画像診断・競合比較はβ対象外 |

## 4. 無料 / 有料の切り分け（β時点）

診断の限界コストが¥0のため、**βは全機能無料で需要を取りに行く**。課金は需要確認後（Phase2）に開放する。

| | β（無料・¥0運営） | 将来の有料（Phase2以降・コスト or 継続価値があるもの） |
|---|---|---|
| 6軸スコア＋改善提案TOP2＋根拠 | ✅ | ✅ |
| ジャンル汎用診断 | ✅ | ✅ |
| ジャンル特化モデル（多シード精緻版） | ― | ✅ |
| サムネ画像診断（Vision＝API課金） | ✕（禁止） | ✅ |
| 競合比較（診断時API quota消費） | ✕（禁止） | ✅ |
| 診断履歴・トレンド・チャンネル適合 | ― | ✅ |
| 回数無制限・一括診断・API連携 | ― | ✅ |

- **βの回数制限:** 限界コスト¥0のため寛容に（例：1ユーザー1日10回程度のソフト上限。乱用検知のみ）。
- **設計原則:** ¥0で出せるもの＝無料、コストが立つもの（Vision・ライブAPI）と蓄積価値＝有料。これでバズっても赤字化しない。

## 5. ユーザーがお金を払うポイント（課金仮説・β後に検証）

人は「採点」には払わない。**「で、どう直せばいいの？」の深い答え**に払う。

1. ジャンル特化の精緻な診断（自分のジャンルでの相対評価）
2. サムネ画像診断（テキストでは不可＝価値が明確）
3. 競合との比較
4. 履歴・トレンド・チャンネル適合（継続価値）

> βではこの仮説を**検証するためのログ**（誰がどのジャンルを何回診断し、再診断したか）を取得することが最重要。

## 6. β公開の成功基準（PM定義）

### 6-0. β核チェックの価値基準（最優先・精度より上位）

> **この商品の価値は「未来を100%予測すること」ではない。投稿前の失敗確率を下げ、改善できる行動を提示することにある。**

技術的な精度（予測の当たり外れ）ではなく、**下の3つが「YES」になるか**をβ核チェックの最上位基準とする。**精度だけを追わない。**

- [ ] **次の行動が分かるか** — 診断結果を見た後、ユーザーが「次に何を直せばいいか」を具体的に分かる
- [ ] **不安が減るか** — 投稿前の「これで伸びるか不安」が、診断後に軽くなる
- [ ] **また使いたいか** — 次に投稿する前にも、また使いたいと思う

> 判定の考え方：スコアが多少ズレていても、上記3つが満たされていれば**商品価値は成立**。逆に精度が高くても3つが「NO」なら価値は無い。

### 6-1. 公開可の技術・運用基準

βは「完成」ではなく「検証」。以下が満たせたら公開可。

- [ ] 投稿前メタデータ（タイトル＋ジャンル必須）だけで診断が走る
- [ ] 総合スコア・6軸・改善提案TOP2・根拠・免責が表示される
- [ ] 診断時に LLM/有料API を一切呼ばない（ログで確認）
- [ ] モデルが各ジャンル最低限のシードで学習済み（初回体験で破綻しない信頼度）
- [ ] 初見ユーザーが説明なしで1回診断を完了できる（相沢CS確認）
- [ ] 利用ログ（ジャンル・回数・再診断）が取得できる（需要検証の計測）

### β後に見る需要シグナル（KPI）
- 週間アクティブ診断ユーザー数 / 再診断率 / ジャンル分布 / 「もう一度使いたい」定性フィードバック
