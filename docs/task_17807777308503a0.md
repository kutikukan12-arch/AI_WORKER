## 1. 目的・背景

### なぜUIが必要か

現在のAI_WORKERにおけるインターフェース状況：

| 対象 | 現在の手段 | 課題 |
|------|-----------|------|
| CEO（社内） | Discord Bot + 自然文コマンド | Discordアプリが必須・モバイル操作が煩雑 |
| 外部ユーザー | YouTube診断AI Webフォーム | 診断1機能のみ・拡張性なし |
| AI社員間 | Discord チャンネル + inbox/outbox ファイル | CEOが手動で貼り付けが必要なフローが残存 |

**本ドキュメントの対象:** 外部ユーザー（βテスター・一般ユーザー）がAI機能を利用するための **Webアプリケーション UI** の設計指針。

## 2. ターゲットユーザー

### プライマリユーザー
- **YouTubeクリエイター（個人）:** 投稿前の動画タイトル最適化に利用
- **中小YouTubeチャンネル運営者:** 複数動画の一括分析・改善サイクルに利用

### セカンダリユーザー
- **社内CEO（検証用）:** AI機能のプロトタイプ確認
- **βテスター（相沢CS管理）:** 機能フィードバック収集

## 4. UI設計方針

### 4.1 設計原則

1. **ゼロ摩擦入力:** 必須項目はタイトルのみ。その他は任意。
2. **即時フィードバック:** 入力中にリアルタイムヒントを表示（将来拡張）。
3. **モバイルファースト:** スマホで操作できること（YouTubeクリエイターの主要デバイス）。
4. **結果の共有可能性:** 診断結果をURLで共有・スクリーンショットしやすいレイアウト。

### 4.2 情報アーキテクチャ

```
/ (トップ)
├── /diagnose          ← メイン診断フォーム
│   └── /result/:id    ← 診断結果表示（共有用URL）
├── /history           ← 診断履歴（認証後のみ・将来）
└── /about             ← サービス説明・フィードバック
```

## 5. 画面設計

### 5.1 診断フォーム画面（`/diagnose`）

**必須入力エリア（上部・目立たせる）**
```
┌─────────────────────────────────────┐
│  動画タイトルを入力してください       │
│  [                              ] │
│  ジャンル: [選択▼]                  │
│                                    │
│  　　　　[診断する →]               │
└─────────────────────────────────────┘
```

**任意入力エリア（折りたたみ式）**
```
▼ 精度を上げる（任意）
  動画時間: [    ] 分
  投稿予定: [日時選択]
  登録者数: [選択▼]
```

**UIコンポーネント要件:**

| コンポーネント | 仕様 |
|-------------|------|
| タイトル入力 | テキストエリア・最大100文字・文字数カウント表示 |
| ジャンル選択 | ドロップダウン・「未選択」デフォルト |
| 診断ボタン | ローディングスピナー付き・二重送信防止 |
| 任意項目 | `<details>` 折りたたみ or アコーディオン |

---

### 5.2 診断結果画面（`/result/:id` または同一ページ下部）

**上部サマリー**
```
┌─────────────────────────────────────┐
│  総合スコア: 72点                    │
│  「タイトルのSEO強化で+8点の余地あり」│
└─────────────────────────────────────┘
```

**6軸レーダーチャート**
```
     タイトル
    SEO  構成
  投稿  差別化
    視聴期待
```

**改善提案リスト**
```
🔴 優先度 高
  → タイトルに検索キーワードを追加してください

🟡 優先度 中
  → 投稿時間を19〜21時にすると視聴率が上がる可能性があります

🟢 優先度 低
  → ...
```

**アクションエリア**
```
[もう一度診断する]  [結果を共有]  [フィードバックを送る]
```

## 6. 技術仕様（実装参照用）

### フロントエンド

| 項目 | 推奨 | 備考 |
|------|------|------|
| フレームワーク | Next.js（App Router）または Vanilla JS | 既存の実装に合わせる |
| スタイリング | Tailwind CSS または CSS Modules | モバイル対応を優先 |
| チャートライブラリ | Chart.js または Recharts | レーダーチャート対応 |
| フォーム管理 | React Hook Form または native | バリデーション必須 |

### バックエンド連携

```
POST /api/diagnose
  Body: { title, genre, duration?, schedule?, subscribers? }
  Response: { score, axes, suggestions, resultId }

GET /api/result/:id
  Response: 診断結果（共有URL用）
```

### セキュリティ要件

| 要件 | 対応方法 |
|------|---------|
| レート制限 | IP単位で1分間10リクエスト上限 |
| XSS対策 | ユーザー入力をサニタイズしてからレンダリング |
| CORS | 許可ドメインを明示設定 |
| ログ | 入力タイトルはログに残さない（PII考慮） |

## 7. UX配慮事項

### エラー状態

| エラー | 表示方法 |
|--------|---------|
| タイトル未入力 | インラインエラー「タイトルを入力してください」 |
| API失敗 | 「しばらく経ってから再試行してください」トースト |
| タイムアウト | スピナー停止 + リトライボタン表示 |

### ローディング状態

- ボタン押下後：スピナー表示 + ボタン無効化
- 結果表示前：スケルトンUI（チャートエリアにプレースホルダー）
- 目標レスポンス：3秒以内（βテスト観測値に基づく）

### アクセシビリティ

- `<label>` と `<input>` の関連付け必須
- ボタンにフォーカスリング
- スコアカラーは色だけに依存しない（アイコン併用）

## 8. 優先実装順序（相沢CS・市川PMとの合意事項）

| 優先度 | 機能 | 理由 |
|--------|------|------|
| P0 | タイトル入力 + 診断 + 結果表示 | βテスト中の最小機能 |
| P1 | 結果共有URL | βユーザーのフィードバック収集を促進 |
| P2 | 診断前後比較（A/Bタイトル） | 「直してもう一度」フローが頻繁に要求される |
| P3 | 診断履歴保存（認証あり） | リピーター向け・βテスト後に検討 |
