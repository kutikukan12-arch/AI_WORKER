# ユーザーインターフェース設計ドキュメント

**対象システム**: AI-Worker Discord Bot  
**作成日**: 2026-05-31  
**バージョン**: Phase 6対応版

---

## 1. システム概要

AI-Worker は Discord をフロントエンドとした AI 自律開発支援システムです。ユーザーはテキストコマンドを通じてタスク登録・実行・承認・管理を行い、Claude Code / OpenAI Codex が自動的にコードを生成・レビュー・GitHub に統合します。

### インターフェース特性

| 項目 | 内容 |
|------|------|
| プラットフォーム | Discord (テキストチャンネル) |
| コマンドプレフィックス | `!` |
| 入力方式 | テキストコマンド |
| 出力方式 | Embed メッセージ・テキスト返信 |
| 対象ユーザー | 開発者 (Bot オーナー = 単一ユーザー想定) |

---

## 2. チャンネル構成設計

ユーザーが操作・確認するチャンネルを目的別に分離します。

```
Discord サーバー
├── #general-work          (ALLOWED_CHANNEL_IDS) ← コマンド入力メイン
├── #ai-review             (AI_REVIEW_CHANNEL_ID) ← AI コードレビュー結果
├── #codex-review          (CODEX_REVIEW_CHANNEL_ID) ← GPT-4 レビュー結果
├── #pull-requests         (PR_CHANNEL_ID) ← PR 作成通知
├── #review-history        (HISTORY_CHANNEL_ID) ← レビュー履歴ログ
├── #batch                 (BATCH_CHANNEL_ID) ← ナイトバッチ結果
├── #meetings              (MEETING_CHANNEL_ID) ← AI チームミーティング
├── #github-log            (GITHUB_LOG_CHANNEL_ID) ← GitHub 操作ログ
└── #errors                (ERROR_CHANNEL_ID) ← エラー通知
```

### チャンネル別ユーザーアクション

| チャンネル | 主な操作 | 通知種別 |
|-----------|---------|---------|
| `#general-work` | コマンド入力・承認・拒否 | 双方向 |
| `#ai-review` | レビュー確認・`!apply-review` | 受信のみ |
| `#pull-requests` | PR 確認・`!create-pr` | 受信のみ |
| `#errors` | エラー確認・対応指示 | 受信のみ |

## 3. コマンドリファレンス

### 3.1 タスク実行

```
!claude <指示内容>
```
Claude Code に実装タスクを依頼します。タスクは自動的に優先度付けされキューに登録されます。

```
!codex <内容>
```
OpenAI GPT-4 によるコードレビューを依頼します。モバイルからの入力にも対応。

```
!run-next
```
キュー内の最高優先度タスクを手動実行します。

---

### 3.2 タスク管理

```
!task list
```
全タスクを優先度順で一覧表示します。

```
!task <タスクID>
```
指定タスクの詳細（状態・優先度・内容・履歴）を表示します。

```
!task add [TYPE] <内容>
```
新規タスクを作成します。

**TYPE オプション:**

| タイプ | 用途 |
|--------|------|
| `IMPLEMENT` | 機能実装 |
| `FIX` | バグ修正 |
| `REFACTOR` | コード改善 |
| `RESEARCH` | 調査・分析 |
| `DOCS` | ドキュメント作成 |
| `TEST` | テスト追加 |
| `REVIEW` | コードレビュー |

```
!task done <タスクID>    # 完了マーク
!task hold <タスクID>    # 保留
!task resume <タスクID>  # 保留解除
!task split <タスクID>   # 大きなタスクを分割
!task merge <タスクID>   # 小タスクを統合
!task stats              # 統計情報表示
!task cleanup            # 完了タスクのクリーンアップ
!task archive            # タスクをアーカイブ
```

```
!queue           # タスクキュー表示
!queue clear     # キューをクリア
```

---

### 3.3 承認ワークフロー

高リスクタスクは自動的に承認待ちになります。

```
!approve [タスクID]    # タスクを承認・実行許可
!deny <タスクID>       # タスクを拒否
!pause <タスクID>      # 実行中タスクを一時停止
!resume <タスクID>     # 一時停止を解除
```

**承認フロー図:**

```
タスク登録
    │
    ▼
リスク判定 ──高リスク──▶ 承認待ち ──!deny──▶ 拒否 (終了)
    │                        │
    │ 低リスク           !approve
    │                        │
    ▼                        ▼
自動実行 ◀───────────── 実行許可
    │
    ▼
AI レビュー → (任意) Codex レビュー → GitHub PR → 完了
```

---

### 3.4 レビュー・PR 操作

```
!apply-review <タスクID>    # Codex フィードバックを自動適用
!create-pr <タスクID>       # GitHub PR を手動作成
!history [タスクID]         # レビュー履歴を表示
!review list                # コードレビュー一覧
!review show <ID>           # レビュー詳細表示
!research list              # リサーチレポート一覧
!research show <ID>         # リサーチ詳細表示
```

---

### 3.5 プロジェクト管理

```
!project current            # 現在のプロジェクトを表示
!project list               # プロジェクト一覧
!project create <名前>      # 新規プロジェクト作成
!project switch <名前>      # プロジェクト切り替え
!project runner on          # Auto Runner を有効化
!project runner off         # Auto Runner を無効化
!project runner status      # Auto Runner の状態確認
```

---

### 3.6 AI チームミーティング

```
!meeting <トピック>          # AI チームで議論 (Claude + Codex + ChatGPT)
!meeting full <トピック>     # 詳細モードで議論
```

複数の AI が協調してアーキテクチャ・設計・方針を議論し、結果を `#meetings` チャンネルに出力します。

---

### 3.7 システム管理

```
!batch      # ナイトバッチを手動実行
!doctor     # システム診断・ヘルスチェック
!help       # コマンド一覧を表示
```

## 4. ユーザー操作フロー

### 4.1 基本的な開発依頼フロー

```
ユーザー: !claude ログイン機能のバグを修正してください

Bot: [タスク登録確認 Embed]
     ┌─────────────────────────────┐
     │ タスク登録完了 #TASK-001    │
     │ 種別: FIX                   │
     │ 優先度: 85 / 100            │
     │ 状態: PENDING               │
     └─────────────────────────────┘

Bot: [実行開始通知]
     Claude Code 実行中... ⏳

Bot: [完了通知]
     ┌─────────────────────────────┐
     │ タスク完了 #TASK-001        │
     │ 実行時間: 2分34秒           │
     │ 変更ファイル: 3件           │
     └─────────────────────────────┘

#ai-review:
     [AI レビュー結果 Embed]
     コード品質: ✅ 良好
     セキュリティ: ⚠️ 軽微な指摘 1件
```

### 4.2 承認が必要なフロー

```
ユーザー: !claude データベースのスキーマを変更してください

Bot: ⚠️ 高リスクタスクを検出
     !approve TASK-002 で承認してください
     !deny TASK-002 で拒否できます

ユーザー: !approve TASK-002

Bot: ✅ 承認されました。実行を開始します。
```

### 4.3 Auto Runner による自律実行フロー

```
ユーザー: !project runner on

Bot: 🤖 Auto Runner を有効化しました
     (キュー内のタスクを自動的に順次実行します)

[自動実行中 - ユーザー操作不要]
↓
タスク1 実行 → レビュー → PR → 完了
↓
タスク2 実行 → 高リスク検出 → 承認待ち
↓
Bot: ⚠️ @オーナー 承認が必要です: #TASK-007
      !approve TASK-007
```

## 5. メッセージ出力設計

### 5.1 Embed カラーコード

| 状態 | カラー | 用途 |
|------|--------|------|
| 成功・完了 | 緑 `#00ff00` | タスク完了・承認 |
| 警告・待機 | 黄 `#ffff00` | 承認待ち・高リスク |
| エラー | 赤 `#ff0000` | 失敗・拒否 |
| 情報 | 青 `#0099ff` | 実行中・一覧表示 |
| ニュートラル | グレー | 履歴・ログ |

### 5.2 メッセージ長の制限

Discord の文字数制限に対応するため、長い出力は自動的に分割されます。

- Embed フィールド: 1024文字以内
- Embed 説明: 4096文字以内
- メッセージ本文: 2000文字以内

長いコード・レビュー結果はファイル添付またはチャンネル別出力で対応します。

## 6. 権限設計

### 6.1 ユーザー権限

現バージョンでは **単一オーナー制** を採用しています。

| 設定変数 | 説明 |
|---------|------|
| `DISCORD_OWNER_ID` | 全コマンドの実行権限を持つユーザー ID |
| `ALLOWED_CHANNEL_IDS` | コマンドを受け付けるチャンネル ID (カンマ区切り) |

### 6.2 セキュリティ制御

- 指定チャンネル以外のコマンドは無視
- オーナー以外のユーザーのコマンドは無視
- タスクタイムアウト (`TASK_TIMEOUT_SECONDS`) による暴走防止
- 同時実行数制限 (`MAX_CONCURRENT_TASKS`)
- 高リスクタスクの手動承認フロー

## 9. エラーメッセージ設計

ユーザーが遭遇する可能性のあるエラーと推奨対応。

| エラー | 表示メッセージ | 推奨対応 |
|--------|--------------|---------|
| タイムアウト | ⏱️ タスクがタイムアウトしました | `!task` で状態確認、`!doctor` で診断 |
| 同時実行上限 | ⚠️ 実行中のタスクが上限に達しています | `!queue` で状況確認、完了を待つ |
| 権限エラー | ❌ このコマンドを実行する権限がありません | オーナー ID 設定を確認 |
| GitHub エラー | ❌ GitHub への接続に失敗しました | `GITHUB_TOKEN` 設定を確認 |
| 不明なタスク ID | ❌ タスク ID が見つかりません | `!task list` で正しい ID を確認 |

## 10. UX 改善ガイドライン

現システムの制約を踏まえた操作性改善の指針。

### 現状の制約
- Discord テキストのみのインターフェースのため視覚的フィードバックが限定的
- 長い出力は読みにくい場合がある

### 改善指針

1. **進捗フィードバック**: 長時間タスクは定期的に「実行中...」の更新メッセージを送信する
2. **コマンド補完ヒント**: 入力ミス時に正しいコマンドを提示する (`!taskk` → `!task を意味しますか？`)
3. **一覧の見やすさ**: `!task list` の出力はページネーション対応 (10件ごとに分割)
4. **タスク ID の短縮**: 長い UUID より短い連番 ID (#001, #002) の優先表示
5. **クイックアクション**: 完了通知に `!approve` / `!deny` ボタンへの誘導テキストを含める
