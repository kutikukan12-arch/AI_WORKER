## 2. Auto Policy によるコマンド振る舞いの変化

### 2.1 ポリシー 4 段階と Discord 表示

`!claude` コマンドを実行すると Auto Policy がタスクを事前評価し、以下の通知が返ります。

| ポリシー | 絵文字 | ユーザーへの表示 | 必要なアクション |
|--------|--------|----------------|----------------|
| `AUTO_SAFE` | 🟢 | `自動実行します` | なし（即時実行） |
| `AI_REVIEW_REQUIRED` | 🟡 | `AIレビュー後に実行します` | なし（自動進行） |
| `HUMAN_APPROVAL_REQUIRED` | 🔴 | `承認が必要です: !approve <ID>` | `!approve <ID>` |
| `BLOCKED` | ⛔ | `この操作は禁止されています` | 依頼内容の変更 |

### 2.2 BLOCKED となる操作例

以下のキーワードを含む依頼は自動的に BLOCKED となりエラー通知が届きます。

- `git push --force` / `git reset --hard`
- `rm -rf` / `del /s /f`
- `.env の表示` / `APIキー を表示`
- `shutdown` / `format C:`

### 2.3 HUMAN_APPROVAL_REQUIRED の操作例

- `.env の変更` / `本番へのデプロイ`
- `PR のマージ` / `Bot の再起動`

## 3. AI Predictor による予測表示

タスク登録時、AI Predictor が自動的に以下の予測を Embed に付加します。

### 3.1 予測表示フォーマット

```
📊 AI 予測
  担当 AI : Claude Code (信頼度 78%)
  成功確率 : 85%
  推定時間 : 約 3〜8 分
  リスク要因: なし
```

### 3.2 担当 AI ルーティング基準

| AI | 得意なタスク種別 |
|----|----------------|
| Claude Code | 実装・新機能追加・コード生成・テスト |
| Codex (GPT-4) | バグ修正・最適化・リファクタリング・セキュリティ |
| ChatGPT | 仕様設計・相談・優先度判断・UI/UX 提案 |

キーワードが複数 AI にマッチする場合は最高スコアの AI が選択されます。

### 3.3 リスク要因による警告表示

以下のキーワードを含む場合、予測表示に警告が追加されます。

| リスク要因 | 成功確率ペナルティ | 表示例 |
|-----------|----------------|--------|
| 認証・機密関連 | -15% | ⚠️ 認証・機密関連 |
| データ削除操作 | -20% | ⚠️ データ削除操作 |
| 本番環境変更 | -10% | ⚠️ 本番環境変更 |
| DB 操作 | -8% | ⚠️ DB操作 |

## 4. Auto Project Runner のユーザーフロー

### 4.1 基本操作

```
!project runner on      # Auto Runner を有効化
!project runner off     # Auto Runner を停止
!project runner status  # 実行状態・統計を表示
```

### 4.2 ステータス表示

`!project runner status` のレスポンス例:

```
🤖 Auto Runner ステータス
  状態      : 実行中 ✅
  ループ数  : 12
  総タスク数: 7 件作成済み
  現フェーズ: Phase-2
  LLM 呼出  : 4 回 (fallback: 1 回)
  推定コスト: $0.020
```

### 4.3 自動一時停止とユーザー通知

Runner が HUMAN_APPROVAL_REQUIRED タスクを検出すると自動停止し、以下の通知が届きます。

```
⚠️ @オーナー Auto Runner が一時停止しました
タスク #TASK-042 の承認が必要です。

👉 !approve TASK-042  → 承認して再開
   !deny TASK-042     → 拒否してキャンセル
```

### 4.4 Auto Resume の通知

安全と判断された一時停止タスクは自動再開されます。再開時には以下の通知が出ます。

```
▶️ タスク #TASK-038 を自動再開しました
  再開理由: 安全な一時停止タスク
```

## 5. コマンド体系（Phase E 更新版）

### 5.1 タスク実行コマンド

| コマンド | 説明 |
|---------|------|
| `!claude <指示>` | タスク登録 → Auto Policy 評価 → 実行 |
| `!codex <内容>` | GPT-4 へのレビュー依頼（モバイル対応） |
| `!run-next` | 最高優先度タスクを手動実行 |
| `!next` | 次のタスク候補を 1 件表示（実行はしない） |

### 5.2 承認コマンド

| コマンド | 説明 |
|---------|------|
| `!approve [タスクID]` | タスク承認（引数なしで一覧表示） |
| `!deny <タスクID>` | タスク却下 |
| `!pause <タスクID>` | 一時停止 |
| `!resume <タスクID>` | 一時停止解除 |

### 5.3 管理コマンド

| コマンド | 説明 |
|---------|------|
| `!queue` | キュー状況表示 |
| `!queue clear` | 待機中タスクをクリア |
| `!task list/stats/done/hold/resume` | タスク管理 |
| `!project runner on/off/status` | Auto Runner 制御 |
| `!meeting <議題>` | AI チーム会議 |
| `!batch` | ナイトバッチ手動実行 |
| `!doctor` | システム診断 |
| `!restart [confirm]` | Bot 再起動（管理者のみ） |
| `!help` | コマンド一覧表示 |

### 6.1 Embed 構造（Phase E 標準）

├──────────────────────────────────────────────┤

│ 実行時間: 3分12秒  変更: 5ファイル            │

│ 🔗 PR: https://github.com/.../pull/42        │  ← オプション
└──────────────────────────────────────────────┘

### 6.2 文字数制限の遵守

| 要素 | 上限 (実用値) | 超過時の処置 |
|------|------------|------------|
| プレーンメッセージ | 1800 文字 | `truncateMarkdown()` で切り詰め |
| Embed 説明 | 3000 文字 | 同上 + `reviews/` へのリンク付加 |
| Embed フィールド | 900 文字 | 同上 |

### 6.3 スマートフォン操作の最適化

`formatter.js` の `formatSmartphoneCommand()` を使用し、コマンドをワンタップでコピーできる形式で提示します。

```
次のコマンドを実行してください:
```txt

```
```

## 8. エラーメッセージ設計（Phase E 更新）

| エラー種別 | 表示メッセージ | 推奨アクション |
|-----------|--------------|--------------|
| BLOCKED 操作 | ⛔ この操作は禁止されています | 依頼内容を変更して再送 |
| 承認タイムアウト | ⏱️ 承認待ちがタイムアウトしました | `!approve <ID>` を再実行 |
| Auto Runner 停止 | ⚠️ Auto Runner が一時停止しました | `!approve <ID>` または `!deny <ID>` |
| LLM 呼び出し失敗 | 🔄 rule-based fallback に切り替えました | 通常通り継続（精度が低下する場合あり） |
| タスクタイムアウト | ⏱️ タスクがタイムアウトしました | `!doctor` でシステム診断 |
| 同時実行上限 | ⚠️ 実行中タスクが上限に達しています | `!queue` で状況確認 |
| GitHub 接続失敗 | ❌ GitHub への接続に失敗しました | `GITHUB_TOKEN` 設定を確認 |
| 不明なタスク ID | ❌ タスク ID が見つかりません | `!task list` で確認 |

## 9. UX 改善指針（Phase E）

### 9.1 透明性の向上

Auto Policy と AI Predictor の判定結果を常にユーザーに開示します。自動実行された理由・承認が必要な理由を明示することで、ユーザーが「なぜこうなったか」を理解できるようにします。

### 9.2 介入ポイントの明確化

自律実行中であっても、ユーザーが介入できるポイントを常に明示します。

```
🤖 Auto Runner 実行中
タスク #TASK-005 を処理しています... (ループ 3/∞)

いつでも停止できます:
  !project runner off    # Runner を停止
  !pause TASK-005        # このタスクを一時停止
```

### 9.3 コスト・リソース透明性

LLM 呼び出し回数と推定コストを `!project runner status` で常時確認できるようにします。過剰な API 利用を防ぐための可視化を提供します。

### 9.4 モバイルファーストな通知設計

- 重要通知（HUMAN_APPROVAL_REQUIRED / BLOCKED）は本文に `@オーナー` メンションを付与
- コマンドは必ずコードブロックで提示（ワンタップコピー）
- 5 行以内に収める（`formatForHuman()` 準拠）
- 長文は `reviews/` ファイルへのリンクで参照
