## 0. 前提（既に在るもの）

実装済みで再利用する資産（`bot/utils/`）:

| ファイル | 役割 | β での扱い |
|---|---|---|
| `youtube-feature-extractor.js`（16次元 `encode`） | メタdata→特徴量 | 再利用（ジャンル/投稿時間の特徴反映が要確認） |
| `youtube-predictor.js`（`diagnosePrePub` / `buildDiagnosisSummary` / `predictPrePub`） | 投稿前診断・ローカル推論・根拠/対策生成 | **正典エンジン。これを仕上げる** |

## 1. 入力（ユーザーが何を入れるか）

**Webアプリの入力フォーム**（URLを開くだけ・ログイン不要・URL不要の投稿前メタデータ）:

```
[ タイトル                          ] ← テキスト入力（必須）
[ ジャンル ▼ ]                        ← セレクト（必須）
（任意・折りたたみ）尺 / 投稿予定時間   ← 最小限。βでは無くてもよい
[ 診断する ]                          ← ボタン
```

| フォーム項目 | 必須 | 備考 |
|---|---|---|
| タイトル | ★必須 | 1行入力・最大100文字 |
| ジャンル | ★必須 | セレクト：VTuber/切り抜き/音楽/実写/ゲーム実況/その他 |
| 尺（秒） | 任意 | 折りたたみ内。例 600=10分 |
| 投稿予定時間 | 任意 | 折りたたみ内。投稿タイミング評価に使用 |

- **必須2項目（タイトル＋ジャンル）だけで診断が走ること。** 肥大化防止のため、タグ・説明文・登録者数等の追加入力欄は**βでは必須にしない**（エンジンは受け付け可だがUIは最小）。
- 必須未入力時はフォーム上でその場バリデーション（落ちない・分かるメッセージ）。
- **viewCount/likeCount/commentCount は入力させない**（投稿前のため。`diagnosePrePub` が既に除外）。

## 2. 処理（既存エンジンをどう使うか）

```
入力パース
  ↓
youtube-feature-extractor.encode(video)   ← 特徴量化（ジャンル/投稿時間も特徴に反映されているか確認）
  ↓
youtube-predictor.diagnosePrePub(video)   ← ローカルモデル推論（LLM/API呼び出しなし）
  → { score(0-100), rank(A/B/C/D), positivePoints[], improvementPoints[] }
  ↓
6軸へのマッピング層（新規・薄い）
  → 各 reason.factor を CTR/維持/SEO/感情/タイミング/差別化 の6軸に割り当て、軸別スコアを算出
  ↓
診断結果オブジェクト（score/rank/6軸/改善提案/免責）を JSON で返す
  ↓
Web フロントが結果画面に整形して表示
```

> エンジン（`diagnosePrePub` / `buildDiagnosisSummary` 相当の結果生成）は**そのまま**。Web側は結果を**受け取って表示するだけ**。`buildDiagnosisSummary` はDiscordのEmbed文字列向けなので、Web用には**構造化JSONを返す薄いアダプタ**を1つ用意する（エンジン本体は変更しない）。

### 実装タスク（差分のみ）

**UI層（今回のWeb化スコープ）:**
1. **Web最小構成**：入力フォーム（タイトル＋ジャンル）→ 診断APIエンドポイント（Nodeサーバが既存エンジンを `require` して `diagnosePrePub()` を呼ぶ）→ 結果JSON → 結果画面表示。社内Discordとは別デプロイ。
   - 1a. 診断結果を返す薄いAPI（既存エンジン流用・LLM/外部API呼び出しなし）
   - 1b. 入力フォーム画面・結果表示画面（6軸/改善提案/免責）
   - 1c. フィードバック取得（簡易・結果画面）
   - 1d. 構造化JSONアダプタ（Embed文字列ではなくJSONで返す）

**エンジン層（既存タスク `task_1780499019570` で進行中・本Web化では触らない）:**
2. **6軸マッピング**：`improvementPoints`/`positivePoints` の factor を6軸へ割り当て、軸別スコアを出す薄い層。factor→軸の対応表を `youtube-predictor.js` 近傍に定義。
3. **ジャンル比較**：feature-extractor がジャンルを特徴に含むか確認。含まなければジャンル係数 or ジャンル別基準値を導入（ローカルのみ・APIなし）。
4. **投稿時間評価**：ジャンル別の視聴者活動時間帯ヒート表（静的JSON）と投稿予定時間を照合してタイミング軸に反映。
5. **改善提案TOP2へ絞り込み**＋**免責文の常時付与**を `buildDiagnosisSummary` に追加。
6. **シード補充（最優先・別働でOK）**：`!youtube collect` を各ジャンルで実行しモデルを現状20件 → 各ジャンル数百件規模へ。`!youtube train` で更新。

### 禁止事項（実装レベル）
- 診断経路から Claude/OpenAI SDK・`fetch` での外部AI呼び出しを**入れない**。
- 診断時に YouTube Data API を**叩かない**（収集はオフライン `!youtube collect` のみ）。
- 出力に再生数の数値・レンジ・「バズる/○万回」等の断定語を**入れない**。

## 3. 出力（ユーザーに何を表示するか）

**Webの結果画面**（エンジンが返すJSONをフロントが整形。表示内容は下記と同じ）:

```
📋 YouTube 投稿前診断レポート

🏆 総合スコア: 72/100  ランク B（良好）

📊 6軸評価
  CTR適性     ★★★☆☆ 65
  視聴維持適性 ★★★★☆ 78
  SEO強度     ★★★☆☆ 60
  感情フック   ★★★★☆ 80
  投稿タイミング ★★☆☆☆ 55
  競合差別化   ★★★★☆ 70

✨ 良い点
  ✅ 感情フック: 冒頭の訴求が強い

🔧 改善点 Top2
  1. CTR適性: タイトルに数字がない
     📌 理由: 数字入りタイトルは注目されやすい傾向
     💡 対策: 「5分でわかる…」のように具体的な数字を入れる
  2. SEO強度: 説明文冒頭にキーワードが薄い
     📌 理由: 冒頭2行のキーワードが検索流入に効きやすい
     💡 対策: タイトルの主要語を説明文の最初の2行に含める

ⓘ 本診断は投稿前の最適化度を示すものです。再生回数を予測・保証するものではありません。
```

- ★評価＋数値＋🟢🟡🔴 等で「ひと目で分かる」こと。
- 改善提案は**最大2件**、各件に**理由＋対策**を必須で併記。
- **免責文を常時フッターに**。
- 結果画面に**簡易フィードバック**（役に立った？＋自由記述1欄）と**URLコピー**だけ置く（SNS連携・共有ボタン群は作らない＝肥大化防止）。

## 4. 成功条件（β公開できる基準）

実装が満たすべき受け入れ条件（守谷CTO 品質レビューのチェック項目も兼ねる）:

- [ ] **Webアプリで** タイトル＋ジャンルの最小入力 → 診断が完了し結果画面が出る（ログイン不要）
- [ ] 総合スコア・ランク・**6軸スコア**・良い点・**改善提案TOP2（理由＋対策）**・**免責**がすべて表示される
- [ ] 結果画面に**簡易フィードバック取得**がある
- [ ] 診断1回の処理で **LLM/有料APIを一切呼ばない**（コードレビュー＋実行ログで確認）
- [ ] 診断時に YouTube Data API を呼ばない（収集はオフラインのみ）
- [ ] 出力に再生数の数値・レンジ・断定語が**含まれない**
- [ ] 必須未入力時にフォーム上で分かるメッセージ（落ちない）
- [ ] モデルが各ジャンル最低限のシードで学習済み（`!youtube status` で sampleCount 確認）
- [ ] **社内 AI_WORKER（Discord）と分離してデプロイされている**（機密が越境しない）
- [ ] 利用ログ（ジャンル/回数/再診断/フィードバック）が取得できる（需要検証の計測）
- [ ] テスト追加：投稿前入力→6軸＋改善提案＋免責が返ることを確認

## 5. スコープ外（β では作らない／CEO確定・肥大化禁止）

**UI/機能の除外（CEO確定）:** ログイン / 決済 / アカウント管理 / 高度な分析画面（ダッシュボード・トレンド）/ SNS連携 / チャンネルOAuth。

**エンジン/将来機能の除外:** サムネVision診断 / 競合比較 / 再生数レンジ / LLM診断 / RAG・VectorDB / チャンネル専用学習 / サブスク決済。

> 提供形態は Web に変更したため「Web UI」はスコープ**内**（最小の入力1画面＋結果1画面のみ）。それ以上の画面・機能追加は禁止。
