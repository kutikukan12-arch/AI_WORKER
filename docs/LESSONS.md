# LESSONS — 再発防止ルール集（会社資産）

> 管理: 🅷 Claude H (Learning / Memory Guardian)
> 目的: 同じ失敗を二度しないAI組織にする。経験を会社資産として扱う。
> 出典: `reviews/`（codexレビュー）・`logs/`（実行ログ）・各プロジェクト `Logs/FixLogs/`
> 初版: 2026-06-01（reviews 137件 / logs 5日分の棚卸し）

各ワーカー・codexレビューは着手前に本ファイルを参照すること。新しい教訓は末尾に `L-xx` で追記する。

---

## 🔴 最優先（高頻度・高コスト）

### L-08 タスクは5分制限前提で SMALL 粒度に分割してから着手する
- **失敗:** タスク失敗の約75%（18/24件）がタイムアウト。`autoSplitOnTimeout` が **343回** 発火。「複雑すぎてタイムアウト」が頻発。
- **ルール:** 着手前に5分で完了する粒度か見積もる。超えそうなら**最初から**分割（事後autoSplitはやり直しコストが大きい）。1タスク=1目的に絞る。

### L-09 完了報告は実コマンド結果で裏取りする（自己申告禁止）
- **失敗:** 依頼「GitHub push が失敗」に対し報告「push 成功」など、実態と矛盾する完了報告が複数。
- **ルール:** 成否は推測で書かない。実際にコマンドを実行し、その出力を根拠に報告する。

### L-10 検証用コマンド出力を報告に添付する
- **失敗:** 完了報告に `git status` / `git log` / push失敗ログが無く、原因特定不能（レビュー指摘 最多）。
- **ルール:** 主張の根拠となる証跡（コマンド出力・ログ全文）を省略しない。

## 🟡 品質・報告

### L-11 修正報告にはコード差分・変更点を明示する
「修正した」だけで差分が無いとレビュー不能。変更ファイル名と該当箇所（diff/前後）を示す。

### L-07 完了報告に動作確認結果表を添える
主要機能を項目化し `✅ / 結果 / 備考` の表で残す（VRChatRankingApp FixLog の確認表が良い型）。

## 🟢 実装

### L-12 分岐ロジックは評価順序に注意する
先行する `return` で後続条件がスキップされるバグに注意。例: 完了判定で `CODE_CHANGE_REQUIRED_TYPES` を `OPS_PROMPT_PATTERNS` より先に評価し、IMPLEMENT タスクの prompt キーワードが無視され誤判定。

### L-13 前提を防御的に確認する
フォルダ/ファイル存在・env 未設定を実行前にチェック。無ければ自動作成 or ユーザーに次アクションを明示するメッセージを出す（`applyFeedback()` のフォルダ欠如エラー由来）。

### L-14 関数の肥大化を避ける
200行級の関数（例 `handleYoutube`）はサブコマンド単位で分割し、可読性・保守性を確保する。

## 📦 ドメイン別（スクレイピング/DB/GUI）

### L-01 スクレイピング先のURL形式を推測しない
対象サイトの実際の形式を確認してから実装。（BOOTH: クエリ形式 `/search?q=` を仮定し404 → 正解はパス形式 `/ja/search/{keyword}`、商品 `/ja/items/{id}`）

### L-02 必要データがHTMLに無ければ代替取得経路を探す
一覧HTMLに欲しい値が無い場合、詳細JSON API等の2段階取得を検討。（ウィッシュリスト数 → `items/{id}.json` の `wish_lists_count`）

### L-03 テストデータを本番DBに残さない
識別可能なID（接頭辞 `T`・1000万超の数値）で投入し、完了後に必ず削除確認。

### L-04 形式変更は既存レコードも一括更新する
URL等の生成形式を変えたら過去データも `UPDATE` で揃える（新旧混在を残さない）。

### L-05 Tkinter: MouseWheel は子ウィジェットに自動伝播しない
スクロール領域を動的生成したら、配下の全ウィジェットへ再帰的に `<MouseWheel>` を再バインド。

### L-06 Tkinter: StringVar 等は参照より前に初期化する
UIビルド関数内で参照する変数は、ビルド呼び出しより前に初期化（初期化順序による AttributeError 防止）。
