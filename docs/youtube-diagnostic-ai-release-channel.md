# YouTube投稿前診断AI — 公開方式の再評価（社内/商品 分離）

- **作成日:** 2026-06-04
- **作成:** 市川（Product Manager）
- **背景（経営方針）:** AI_WORKER Discord の外部公開は**却下**。Discordは社内運営基盤で、開発ログ・AI社員指示・意思決定・商品戦略・失敗履歴・内部改善情報を含むため。
- **今後の不変ルール:** **AI_WORKER = 社内システム ／ 商品 = 外部公開。両者を完全分離する。**
- **関連:** [β版MVP確定仕様](./youtube-diagnostic-ai-beta-mvp.md) / [CFOコスト判断](./youtube-diagnostic-ai-cfo-cost-framework.md) / [商品整合メモ](./youtube-diagnostic-ai-product-alignment.md)

---

## 0. 大前提：エンジンは分離できる（重要）

診断の中核（`youtube-predictor.js` / `youtube-feature-extractor.js` / `youtube-model.json`）は **Discordに依存しない純粋なローカル推論ロジック**。よって：

- **社内（AI_WORKER）**：開発・検証用に Discord から呼び続ける（`!diag` 等）。機密はここに留まる。
- **商品（外部）**：上記エンジンを**共有モジュールとして切り出し**、外部向けフロント（Web等）に載せる。**出すのは「学習済みモデル＋エンジン」だけで、ログ・戦略・失敗履歴は一切持ち出さない。**
- シード収集（`!youtube collect`）は**社内のバッチ作業**のまま。生成された `youtube-model.json` を商品側へ配布する。

> つまり「分離」は技術的に容易。宮城Leadの 6軸モデル接続修正は**フロントに関係なく価値が残る**（presentation層だけ載せ替え）。**開発を止める必要はない。**

## 1. 公開方式の評価

評価基準：①ユーザー導入の簡単さ ②開発コスト ③運用費 ④収益化しやすさ ⑤将来拡張性（◎>○>△）

| 方式 | ①導入 | ②開発 | ③運用費 | ④収益化 | ⑤拡張性 | 総評 |
|------|:--:|:--:|:--:|:--:|:--:|---|
| **A. Webアプリ** | ◎ | ○ | △〜○ | ◎ | ◎ | **本命**。URLを開くだけ・スマホ可・リンク拡散・決済容易 |
| **B. Windowsアプリ** | △ | △ | ◎ | △ | △ | 運用費は最小だが導入と収益化が弱い・Win限定 |
| **C1. Chrome拡張** | △ | △ | ○ | △ | ○ | YouTube Studioに重ねる差別化は強いが審査/開発重・将来候補 |
| **C2. LINEミニアプリ** | ◎ | △ | ○ | ○ | △ | 日本ユーザー導入は良いがUI制約・拡張頭打ち |

### 各方式の詳細

#### A. Webアプリ（PWA対応）— ★推奨
- **①導入 ◎**：インストール不要、URLを開くだけ。スマホ/PC両対応。リンクで共有でき拡散しやすい（CSテストの「共有意向」とも噛み合う）。
- **②開発 ○**：既存Nodeエンジンを**薄いAPIで包む**だけで中核は再利用。フロントは入力フォーム＋結果表示の軽量SPA。
- **③運用費 △〜○**：常時稼働サーバが要る（が、診断はローカル推論＝**変動費¥0**）。小型VPS／サーバレスで小額。後述。
- **④収益化 ◎**：Stripe等でサブスク/従量課金を最短で組める。Web課金の王道。アカウント・無料/有料の出し分けが容易。
- **⑤拡張性 ◎**：アカウント・履歴・サムネアップロード・ダッシュボード・SEO流入・将来のVision/LLM/競合機能、すべて載せやすい。

#### B. Windowsアプリ
- ①△ ダウンロード→インストール→SmartScreen警告。Mac/スマホ不可（配信者にMac層あり）。
- ②△ Electron等＋コード署名＋自動更新の仕組み。
- ③◎ クライアント実行でサーバ最小。モデル同梱なら**完全ローカル¥0運用**も可能。
- ④△ ライセンス/アプリ内課金は手間。決済が弱く、解約・更新管理が重い。
- ⑤△ クロスプラットフォーム弱く、結局Web版と二重メンテになりがち。
- → **オフライン重視/ヘビーユーザー向けの後発オプション**としてはあり。初手ではない。

#### C. その他
- **C1 Chrome拡張**：YouTube Studioの投稿画面に診断を重ねられれば体験は最強。ただしWebストア審査・開発重・Chrome限定。**Web版が立ち上がった後の差別化機能**として有力。
- **C2 LINEミニアプリ**：日本の投稿者層に導入障壁が低い。ただしUI/拡張に制約。Web版のサブ入口として検討余地。

---

## 2. 推奨：**A. Webアプリ（PWA）を第一候補**

理由：分離方針に最も素直（社内Discordと完全に切り離せる）／導入が最も簡単で需要検証に最適／収益化と将来拡張が最強。Bの「運用費最小」も魅力だが、**β＝需要確認フェーズでは「導入の簡単さ」と「拡散しやすさ」が最優先**であり、そこはWebが圧倒。

### 段階リリース案
```
β（需要確認）: 最小Webアプリ
  ・入力フォーム(タイトル+ジャンル必須) → 診断結果(6軸+改善提案+免責)
  ・ログイン任意 or 簡易（メールのみ）。決済なし・全機能無料
  ・利用ログ(ジャンル/回数/再診断)を計測 ← CSテスト/需要KPIの基盤
   ↓
Phase2（需要確認後）: アカウント＋決済（Stripe）＋有料機能（Vision/LLM等）
   ↓
将来: Chrome拡張(Studio連携) / Windowsアプリ(オフライン) を選択的に追加
```

## 3. アーキテクチャ（社内/商品の分離図）

```
[社内 AI_WORKER（Discord・機密）]            [商品（外部公開・機密ゼロ）]
  ・!youtube collect（シード収集/無料枠）        Webフロント(入力フォーム/結果表示)
  ・!youtube train（モデル学習）                      ↓ HTTPS
  ・開発/検証用 !diag                            診断API（Nodeサーバ・薄い）
        │                                              ↓ require
        │  生成物のみ受け渡し                    共有エンジン（predictor/feature-extractor）
        └─────  youtube-model.json  ───────▶      ＋ youtube-model.json（学習済み・配布物）
```
- **越境するのは「学習済みモデル＋エンジンコード」だけ。** ログ・指示・戦略・失敗履歴は社内に留まる。
- 商品リポジトリ／デプロイ環境は AI_WORKER と**別管理**（混在させない）。

## 4. コストへの影響（CFO資料の更新点）

[CFOコスト判断フレーム](./youtube-diagnostic-ai-cfo-cost-framework.md) の前提が「Discord公開＝サーバ不要」から「Webアプリ＝小型サーバ必要」へ変わる。更新点：

| 項目 | 旧（Discord） | 新（Webアプリ） |
|---|---|---|
| 診断の変動費 | ¥0 | **¥0（不変）**※ローカル推論のまま |
| 固定費 | 既存Bot内＝実質0 | **小型VPS/サーバレスのホスティング（小額・月数百〜数千円）** |
| 規模10,000時 | ホスティング小額増 | 同左＋必要ならインスタンス増（それでも小額） |

> 結論は変わらず：**診断は変動費¥0でバズっても赤字化しない。** 追加で「Webホスティングの固定費（小額）」が乗るだけ。CFOのシーリング額判断に、このホスティング月額上限を含めて確認いただく。
