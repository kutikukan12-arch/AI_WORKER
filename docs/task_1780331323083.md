## 1. 概要

AI_WORKER の予測・診断 AI は以下の 2 モデルで構成される。

| モデル | 方式 | 主な用途 |
|--------|------|---------|
| **YouTube予測AI（MLモデル）** | ロジスティック回帰 + ルールベース混合 | 投稿後の再生数ヒット/ミス予測 |
| **YouTube診断AI（LLMモデル）** | Claude API (Sonnet) + RAG | 投稿前の6軸コンテンツ診断 |

本手順書では **YouTube予測AI の MLモデル** のトレーニング・評価を中心に記載する。  
診断AI の Phase2/3 学習（シードプロファイル構築）については末尾の補足を参照。

## 2. モデル構成

### 2.1 MLモデル（ロジスティック回帰）

**特徴量（16次元）:**

| 特徴量 | 説明 | 正規化方法 |
|--------|------|-----------|
| `like_ratio` | いいね数 / 再生数 | そのまま |
| `comment_ratio` | コメント数 / 再生数 × 10 | そのまま |
| `title_len_norm` | タイトル文字数 / 100 | 0〜1 |
| `title_has_excl` | 「!」「！」の有無 | 0 or 1 |
| `title_has_quest` | 「?」「？」の有無 | 0 or 1 |
| `title_emoji_norm` | 絵文字数 / 5 | 0〜1 |
| `title_caps_ratio` | 全大文字単語の割合 | 0〜1 |
| `tag_count_norm` | タグ数 / 30 | 0〜1 |
| `desc_len_norm` | 説明文文字数 / 2000 | 0〜1 |
| `duration_norm` | 動画尺（秒）/ 3600 | 0〜1 |
| `published_hour_sin/cos` | 投稿時刻の周期エンコード（2次元） | −1〜1 |
| `published_dow_sin/cos` | 投稿曜日の周期エンコード（2次元） | −1〜1 |
| `sub_magnitude_norm` | log10(登録者数+1) / 8 | 0〜1 |
| `bias` | 定数項 | 1.0 固定 |

**ラベル定義:**

| ラベル | 条件 | 意味 |
|--------|------|------|
| `hit` | buzz_ratio > 5.0 | 登録者数の5倍以上の再生 |
| `miss` | buzz_ratio < 0.3 | 登録者数の0.3倍未満の再生 |

`buzz_ratio = viewCount / subscriberCount`

### 2.2 混合スコア計算式

```
mlWeight = min(mlSamples / 100, 0.7)
probability = mlScore × mlWeight + ruleScore × (1 − mlWeight)
```

| 学習データ数 | MLの重み | 備考 |
|-------------|---------|------|
| < 20件 | 0% | ルールベース100% |
| 20〜100件 | 14〜70% | 段階的に増加 |
| ≥ 100件 | 70% | 上限固定 |

## 3. トレーニング手順

### Step 1: 環境確認

```
# .env に YOUTUBE_API_KEY が設定されているか確認
!youtube status
```

確認項目:
- `YOUTUBE_API_KEY` が設定済みであること
- 当日のAPIクォータに余裕があること（1日 10,000 units 上限）

### Step 2: シードデータ収集

```
# VTuber ジャンル
!youtube collect vtuber VTuber 歌ってみた

# ゲーム実況ジャンル
!youtube collect gaming ゲーム実況 神回

# その他ジャンル
!youtube collect music 歌ってみた オリジナル曲
!youtube collect clip 切り抜き VTuber
```

**収集量の目安:**

| クォータ消費 | 収集件数 |
|------------|---------|
| 100 units | hit最大100件（検索1回）|
| 100 units | miss最大20件（検索1回）|
| 1〜2 units | 動画詳細取得（videos.list） |

1ジャンルあたり約 200〜600 units 消費する。  
全ジャンルを1日で収集すると 1,000〜3,000 units 程度。

**保存先:** `data/youtube-seeds/<genre>.json`

### Step 3: モデル訓練

```
!youtube train
```

- 全ジャンルの `data/youtube-seeds/` を読み込み、統合して訓練する
- 最低 10 件のサンプルが必要（hit/miss合計）
- MLが有効になるのは 20 件以上から
- 訓練結果は `data/youtube-model.json` に保存される

**モデルファイル構造:**

```json
{
  "weights": [/* 16次元の重みベクトル */],
  "sampleCount": 52,
  "hitCount": 38,
  "missCount": 14,
  "trainedAt": "2026-06-01T12:00:00.000Z",
  "version": "1.0"
}
```

### Step 4: 訓練結果の確認

```
!youtube status
```

確認項目:
- `sampleCount` が期待値か（前回より増えているか）
- `trainedAt` が最新日時か
- ML モデルが有効表示になっているか

## 4. 評価手順

### 4.1 定量評価（`buzz_ratio` 基準）

**評価用サンプルの用意:**

シードデータの 20% をホールドアウト（訓練から除外）して評価に使う。  
Discord コマンドには自動分割機能がないため、手動で評価データを作成する場合は:

```
# 評価したい動画URLで個別予測を実行し、実際のbuzz_ratioと比較
!youtube predict https://www.youtube.com/watch?v=xxxxxxxxxx
```

**評価指標の基準値:**

| 指標 | 目標値 | 備考 |
|------|--------|------|
| ヒット適合率（Precision） | ≥ 0.70 | `hit` と予測した中の実際の hit 割合 |
| ヒット再現率（Recall） | ≥ 0.60 | 実際の hit の中で正しく検出できた割合 |
| 全体正解率（Accuracy） | ≥ 0.65 | hit + miss の総合正解率 |

### 4.2 定性評価（LLM診断精度確認）

診断AIの出力品質を以下の観点で確認する:

| 評価軸 | 確認ポイント | 基準 |
|--------|------------|------|
| CTR予測 | タイトル・サムネへの言及が具体的か | 改善提案が実行可能なレベル |
| SEO強度 | キーワード配置の具体的指摘があるか | NG例/OK例を含むか |
| 感情フック | 冒頭のフック評価が論理的か | 理由の説明があるか |
| 投稿タイミング | ジャンル特性に基づく提案か | 「土日夜がいい」以上の粒度 |

**評価に使う代表入力（投稿前入力 = viewCount = 0）:**

```
タイトル: 【歌ってみた】夜に駆ける / 白詠
説明文: 初めての歌ってみた投稿です。
タグ: 歌ってみた, VTuber, 夜に駆ける
ジャンル: VTuber
viewCount: 0  ← 投稿前なので必ず0
```

> **注意（L-15）:** 投稿前入力で `buzz_ratio = 0` になるため、ルールベーススコアは常に MISS を返す。  
> この状態はバグではなく仕様。投稿前診断は LLM 診断 AI を使うこと。

### 4.3 評価実施タイミング

| タイミング | 実施内容 |
|-----------|---------|
| 初回トレーニング後 | 定量評価 + 定性評価を実施 |
| シードデータ追加後 | `!youtube train` 再実行 + 定量評価 |
| 月次バッチ後 | 全評価項目の確認 |
| L-15 違反検知時 | 代表入力テストを必ず実施 |

## 5. 品質基準

### 5.1 トレーニング完了判定条件

以下を**すべて満たした場合のみ**完了とする:

- [ ] `data/youtube-model.json` が更新されている（`trainedAt` が最新）
- [ ] `sampleCount ≥ 20`（ML有効の最低ライン）
- [ ] `!youtube status` でモデル有効が確認できる
- [ ] ヒット精度 Precision ≥ 0.70 または手動評価で問題なし

### 5.2 再訓練が必要な状態

以下のいずれかに該当した場合は再収集・再訓練を行う:

| 症状 | 判断基準 | 対処 |
|------|---------|------|
| `unknown` が多発 | buzz_ratio 中間帯の動画が多い | シードデータを増やして再訓練 |
| 全て `hit` になる | hit データが偏っている | miss データを追加収集 |
| 全て `miss` になる | miss データが偏っている | hit データを追加収集 |
| 信頼度が `low` のまま | sampleCount < 20 | `!youtube collect` で追加収集 |

## 6. 改善サイクル

### 6.1 月次改善フロー

```
月初: シードデータ追加収集
  ↓ !youtube collect <genre> <query>
データ確認（hit/miss バランス確認）
  ↓ data/youtube-seeds/<genre>.json を確認
モデル再訓練
  ↓ !youtube train
評価実施（代表入力テスト）
  ↓ !youtube predict <url>
品質基準チェック
  ↓ Precision ≥ 0.70, Recall ≥ 0.60
完了報告 / 問題があれば次Stepへ
```

### 6.2 精度向上のための追加データ戦略

**ヒット率が低い場合:**

```
# ヒット動画を追加収集（再生数でソート）
!youtube collect <genre> <ヒットしやすいキーワード>
```

**ジャンル偏りがある場合:**

```
# 弱いジャンルを重点的に収集
!youtube collect gaming ゲーム 縛りプレイ
!youtube collect gaming ゲーム 初見プレイ
```

### 6.3 Phase2 診断AI のジャンルプロファイル更新

診断AI のシード学習（Phase2）は以下のフローで更新する:

```
YouTube Data API でジャンル別シード収集
  ↓ data/diagnostic/seeds/{genre}/hit/*.json
  ↓ data/diagnostic/seeds/{genre}/miss/*.json
Claude API でパターン抽出
  ↓ data/diagnostic/genre/{name}.json
Vector DB に upsert
  ↓ namespace: "genre_profiles"
```

詳細は [診断AIアーキテクチャ設計書 §8](./youtube-diagnostic-ai-architecture.md#8-学習方法設計) を参照。

## 7. トラブルシューティング

### 訓練サンプルが足りない

```
# 各ジャンルを追加収集してから再訓練
!youtube collect vtuber VTuber コラボ
!youtube train
```

### クォータ超過で収集できない

翌日 09:00 JST（UTC 00:00）にリセットされるまで待つ。  
`!youtube status` で当日使用量を確認する。

### モデルが更新されない

`data/youtube-model.json` の `trainedAt` を確認する。  
古い場合は `!youtube train` を再度実行する。

### 登録者数非公開チャンネルの動画が miss ばかりになる

登録者数非公開の場合 `buzz_ratio` が計算できずスコア 0.5（unknown）になる。  
`!youtube collect` の段階でそのチャンネルはスキップされるため、シードデータへの混入は発生しない。  
`!youtube predict` で個別に渡した場合は `信頼度: low` と表示される。
