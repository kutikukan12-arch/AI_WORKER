# Incident / Lesson 対応モデル（= AI_WORKER 標準改善フロー）

> 管理: 🅷 Claude H (Learning / Memory Guardian)
> 制定: 2026-06-04（育野 Learning Manager 指示）
> **位置づけ: AI_WORKER 標準改善フローとして正式採用（2026-06-04）。今後 Incident 発生時はこのモデルを基本形とする。**
> 目的: インシデントや学び候補を「個別対応」で終わらせず、**会社ルールと実行可能な手順**に変えて再発を防ぐ。

今後すべての Incident / Lesson 候補は、原則この7ステップで処理する。

| # | ステップ | 内容 | 成果物 |
|---|---|---|---|
| 1 | **検知 Detect** | 失敗・インシデント・学び候補を受領。背景と事象を整理 | 事象メモ |
| 2 | **判断 Judge** | Lesson化の価値を判定（再発性・横展開性・実害の有無） | 登録可否の判断 |
| 3 | **標準化 Standardize** | ルール化（`docs/LESSONS.md` の `L-xx`）。可能なら**実行可能な手順/スクリプト**にする | LESSONS追記 ＋ npm script |
| 4 | **実チェック Verify** | 手順を**実環境で実行**し、理論でなく実害を検出する（理屈で終わらせない） | 検出結果 |
| 5 | **是正のタスク化 Remediate** | 発見した問題を個別修正で終わらせず、**担当を割り当ててタスク起票**（受け入れ基準付き） | タスク（FIX/RESEARCH） |
| 6 | **資産化 Asset** | LESSONS / スクリプト / メモリを会社資産として commit・同期 | git commit |
| 7 | **申し送り Handoff** | 将来の自動化・正式採用候補を記録し、次工程へ引き継ぐ | 申し送りメモ |

## このモデルの原則（今回の VRChat/.gitignore 事案で確立）
- **「書いた＝安全」と判定しない。** ルールやガードを追加したら、必ず実環境で確認する（例: `.gitignore` 追加だけでは追跡済みファイルは保護されない → L-16）。
- **問題発見を個別修正で終わらせない。** 1件の修正で閉じず、同種を一括検出できる**標準手順（スクリプト）**に昇格させ、会社ルール化する。
- **完成・完了の判定は受け入れ基準と代表入力のE2Eで行う**（L-15）。タスク完了率では判断しない。

## 関連資産
- `docs/LESSONS.md` … 再発防止ルール集（L-xx）
- `scripts/security-check.js`（`npm run security-check`）… L-16 公開境界の標準チェック
- `scripts/lessons-patrol.js`（`npm run patrol`）… reviews/logs の週次学習巡回

## 今後の採用候補（保持）— System Health Manager 構築時に正式採用候補（育野指示・2026-06-04）
1. `npm run security-check` の **定期巡回**
2. **重要変更後チェック**（security-check を変更後に必ず実行）
3. **健康診断 Report**（巡回・チェック結果をまとめた定期ヘルスレポート）
